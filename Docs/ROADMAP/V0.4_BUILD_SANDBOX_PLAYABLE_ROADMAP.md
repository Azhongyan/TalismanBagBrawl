# 《符箓背包》V0.4 BuildSandbox Playable Preview Roadmap

## 1. 阶段定位

```text
V0.4 BuildSandbox Phase 2 = 中层题库规则 + 下层可玩验证舱
```

前置 Phase 1 已完成的是上层发动机：

```text
Build / 标签 / 羁绊 / 多格占位 / 词条品质 / Modifier / Event / Benchmark / 验证池 / 报告
```

Phase 2 要补齐：

```text
地图规则怎么出题
敌人怎么暴露 Build 缺口
Boss 怎么成为多钥匙锁
3-10 / 4-10 如何作为 devOnly 验证章节
用户如何在独立沙盒场景中实际看、点、拖、验证
```

## 1.1 UI 复用与数据面板核心原则

Phase 2 的 UI 原则：

```text
能复用既有 UI 语言就尽量复用。
不要为了每个机制新画很多 UI 框。
机制、调参、题库、Readiness、DropBias、反馈规则都应集中进统一数据面板。
玩家侧只看到线索、反馈和氛围提示，不直接看到答案。
开发者侧数据面板可以看到完整规则和调参项。
```

UI 复用的权威登记表：

```text
Docs/V0.4/UI_REUSE_SOURCE_REGISTRY.md
```

成熟组件扩展规则：

```text
Docs/V0.4/REUSABLE_COMPONENT_EXTENSION_RULE.md
```

最新 Guard 收口：

```text
V0.4 沙盒只验证规则 / 数据 / 构筑逻辑。
成熟 UI / 动画 / 交互手感默认复用 V0.2 / V0.3。
任务窗口不得再仿制一套看起来类似的 UI。
成熟组件不是不能改，而是作为 Base Component 扩展。
新玩法优先新增 Extension Layer / Adapter Layer，不重建职责重叠组件。
```

应优先复用 / 参考：

```text
现有战斗提示
道具造成伤害提示
战斗日志 / 反馈文本
Boss 信息展示语言
状态提示 / Tooltip
Reward / Result / Info 面板的表现习惯
```

禁止默认新增一堆独立框：

```text
每个 Boss 一个新框
每个地图机制一个新框
每种反馈一个新框
每个 DropBias 一个新框
每个 Readiness 项一个新框
```

所有新面板必须优先进入：

```text
BuildSandbox Data Panel / 调参数据面板
```

V04 独立沙盒场景中为了验证逻辑临时创建的 Board / ItemTray / Popup / ProblemSelector / DataBlock，默认只是：

```text
temporary preview / logic test bench
```

不得被当作正式 UI 真源继续精修。后续正式手感必须走：

```text
V0.4 BuildSandbox 数据
→ Adapter / ViewModel
→ V0.2 / V0.3 已登记成熟 UI / 交互源
```

V0.4 新功能示例：

```text
多格占用不是做 V04Board，而是在成熟棋盘上加 BoardOccupancyExtension。
道具形状不是做 V04ItemTray，而是在成熟道具栏上加 ItemTrayShapeExtension。
旋转放置不是做 V04DragSystem，而是在成熟拖拽手感上加 DragRotationPlacementExtension。
Build 字段不是做 V04ItemInfoPopup，而是在成熟信息弹窗上加 ItemInfoBuildFieldsExtension。
```

并要求配置 / 报告 / 数据面板字段层建立中文名 + English stable key：

```text
中文显示名：给用户/策划在数据面板中直接理解。
English stable key：给配置、报告、后续开发稳定引用。
玩家 UI 不需要中英文双语，只显示中文即可。
```

Unity 命名规则：

```text
Hierarchy / GameObject / 文件名 / 脚本 key 使用英文稳定命名。
中文只用于 UI 显示文案、配置 displayName、报告说明、开发者数据面板中文显示名。
不得在 Hierarchy 中新增中文物体名。
```

三层合起来才是完整 V0.4 Build 验证系统：

```text
上层：Build 词条 / 羁绊 / 多格 / 模拟器
中层：地图规则 / 敌人题目 / Boss 解题 / 配置面板
下层：独立可玩沙盒场景 / 数据面板 / 拖拽体验 / 验证流程
```

## 1.2 Stable Battle Runtime 与 BuildSandbox 集成策略

权威策略文件：

```text
Docs/V0.4/BUILD_SANDBOX_INTEGRATION_STRATEGY.md
```

V0.4 与 V0.2 / V0.3 的关系不是二选一：

```text
稳定主链路保留。
V0.4 新能力隔离开发。
纯逻辑模块干净设计。
表现层在 V0.4 沙盒验证。
最终通过 Snapshot / Adapter 合入。
```

当前不做：

```text
不复制第二套正式战斗系统。
不为了 V0.4 强行重构 V0.2 / V0.3 稳定战斗。
不继续把 V0.4 多格摆放硬塞进旧 BattlePrepare 单格拖拽内核。
```

当前 V0.4 多格摆放验证路线：

```text
Scene_TalismanBag_V04_BattleSandboxPreview
→ ShapePlacementSession
→ ShapeAwareItemTrayGrid
→ ShapeGridReceiver
→ MobileShapePlacementInput
```

视觉表层可以尽量复刻 V0.2 / V0.3 成熟 BattlePrepare，交互内核必须使用 V0.4 ShapePlacement。

一句话：

```text
复制皮，换内核。
```

GUARD_KEEP_PHASE2_DEVONLY_AND_ISOLATED

```

含义：

不打开正式 FeatureFlag。
不改 V0.2 / V0.3 正式流程。
```

## 3. 阶段目标

Phase 2 目标不是正式上线 3-10 / 4-10，而是建立 devOnly 可验证版本：

```text
玩家看到题目
→ 判断需要什么 Build
→ 调整背包 / 羁绊 / 词条 / 摆放 / 供能
→ 沙盒模拟验证
→ 看到 Readiness / 失败原因 / 推荐补法
→ 形成可晋升候选
```

## 4. 独立场景原则

可玩验证必须接到独立场景：

```text
Scene_TalismanBag_V04_BattleSandboxPreview
```

禁止接到：

```text
Scene_TalismanBag_V02_FormationCounter
Scene_TalismanBag_V03_MainHome
Scene_TalismanBag_V03_TalismanUpgrade
```

正确结构：

```text
正式 BattleScene / BattlePage
        ↓ 只读参考
BattlePageViewAdapter
        ↓
V04 BattleSandboxPreviewScene
        ↓
BuildSandbox 数据 / 题库 / 模拟器 / Boss 验证池
```

## 5. 配置面板原则

允许新增 V0.4 devOnly 配置面板：

```text
Tools / Talisman Bag / V0.4 / BuildSandbox / Data / [Manual Only] BuildSandbox Config Panel 01
```

该面板可以参考 `Stage Config Panel 01` 的工具形态：

```text
EditorWindow
Tab 页
Foldout
Reload / Validate / Save
Validation 页
ScriptableObject 编辑
```

但数据源必须独立，不得直接编辑 V0.2 / V0.3 正式配置：

```text
不直接编辑 V02RunConfig
不直接编辑正式 EnemyDefinition
不直接编辑正式 DropTable
不直接编辑正式 BossConfig
不直接改 1-10 / 2-10
```

## 7. Phase 2 完成后

Phase 2 通过后不得自动合入正式主线。

只能输出：

```text
PromoteCandidateDraft
```

后续如果要正式合并，必须新开：

```text
V0.4-PromoteToMainline01
```

正式合入顺序必须分层：

```text
1. 先合纯配置候选，不开正式入口
2. 再合只读展示
3. 再开 dev/debug FeatureFlag
4. 再接战斗效果
5. 最后才考虑正式 3-10 / 4-10 章节
```

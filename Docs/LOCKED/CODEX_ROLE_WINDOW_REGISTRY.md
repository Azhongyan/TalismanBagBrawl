# CODEX ROLE WINDOW REGISTRY

本文件定义《符箓背包》Codex 多窗口工作流中的角色窗口注册规则。

它用于解决一个关键边界：记忆文件可以规定流程，但不能凭空完成跨窗口通信。只有角色窗口已注册、当前窗口具备真实线程读取 / 发送能力，并且消息确实发送成功时，才可称为“自动流转”。

## 1. 注册原则

- 角色窗口必须有明确角色、明确窗口标题或线程标识、明确职责范围。
- 角色窗口未注册时，不得假装已经调用该角色。
- 当前窗口无法读取或发送到目标角色窗口时，不得自行代理目标角色完成工作。
- 子代理、临时代理或当前窗口代行分析，不自动取得固定角色身份。
- 无法自动流转时，只能生成清晰报告交给用户手动转发。
- 角色注册表属于项目记忆文件，只能由唯一记忆治理窗口在用户明确审批后更新。
- 其他窗口不得自行修改本文件，不得通过声明“我是某角色”来获得注册身份。

## 2.1 跨系统执行体类别

当前状态：`DISABLED / 暂停生效`。

用户已要求先禁用 Cross-System Executor 协议但不删除文件。未重新获得用户明确审批前，以下类别定义仅作为历史草案保留，不产生正式承接权限。

`Cross-System Executor / 跨系统执行体` 不是固定角色窗口，不登记线程标识，不获得治理权。

当前试运行对象：`ZCode`。

ZCode 只在用户明确指定时，可临时承接：

- 当前版本开发窗口 / 开发执行体：只执行已 `GUARD_PASS` 且用户已审批的任务档案或修复任务档案。
- 外部 QA 执行体：只按正式任务档案验收并落盘 QA 报告。
- 已审批修复包的修复执行体：只执行已经由 Codex tech architect / codex task writer / guard-agents / 用户审批完成的修复任务。

ZCode 不可承接：

- guard-agents
- producer tech pm
- tech architect
- codex task writer
- RepoOps 主责

跨系统执行体的详细边界、交接包、commit 规则、试运行范围与回滚规则见 `Docs/LOCKED/CROSS_SYSTEM_EXECUTOR_PROTOCOL.md`；但当该协议状态为 `DISABLED` 时，ZCode 不得以跨系统执行体身份承接正式任务。

## 3. 既有窗口历史与现行记忆的合并规则

- 已注册的四个固定角色窗口继续保留用户此前上传的记忆档案、开发历史和角色上下文，不要求清空或重建窗口。
- 这些历史内容用于帮助角色理解项目沿革、旧版本决策和长期职责。
- 每次接收新任务时，固定角色仍必须重新读取真实工程中的当前 `AGENTS.md` 与 `Docs/LOCKED/*`。
- 若窗口旧历史、旧附件或旧版守则与当前锁定文档冲突，以真实工程中最新锁定文档为准。
- 合并角色不等于合并权限：producer tech pm、tech architect、codex task writer 与 QA reviewer 仍不得修改项目记忆文件；guard-agents 仍不得直接开发。
- 包体临时窗口不得因为引用了固定窗口历史而获得固定角色身份。

以下是规范化推荐标题，仅用于未来新项目或窗口丢失后的重建，不要求对当前已注册窗口重命名：

固定窗口创建或替换后，必须回到本长期置顶记忆治理窗口登记线程标识。仅标题相近但线程标识未登记的窗口不得冒充固定角色。

## 5. 自动流转成立条件

只有同时满足以下条件，某窗口才可以执行跨窗口自动流转：

1. 目标角色已在本文件中注册。
2. 当前窗口具备真实读取或发送目标线程的工具能力。
3. 当前窗口已经确认目标线程标题 / 标识匹配注册信息。
4. 当前窗口确实完成发送或读取，并在回复中说明结果。
5. 目标角色返回了对应产物，例如失败归因、任务档案或 QA 报告。
6. 返回产物的来源与角色注册信息一致，不是未标注来源的临时草案。

缺任一条件，均不允许声称“已调用 tech architect / codex task writer / QA reviewer”。

仅完成消息发送不代表该阶段完成；必须等待目标角色返回正式产物。

## 6. 正式产物与临时草案

- 已注册固定角色窗口真实返回的职责内产物，才可标记为正式角色产物。
- 子代理可以用于调查、列疑点、整理材料或起草建议，但必须标记为“临时分析草案”。
- 临时分析草案不得直接恢复开发权限，不得替代 architect 失败归因、task writer 修复任务档案或 Guard 回执。
- 用户明确授权某次临时代行，只授权该次辅助工作；除非本注册表随后由记忆治理窗口更新，否则不产生长期角色注册。
- 当 `CROSS_SYSTEM_EXECUTOR_PROTOCOL.md` 状态为 `ENABLED` 时，ZCode 作为跨系统执行体的开发回执或 QA 报告必须通过协议规定的交接包落盘，并经用户确认后，才可作为跨系统正式交接产物被 Codex 读取；它仍不得替代 tech architect、codex task writer 或 guard-agents 的正式产物。当前协议为 `DISABLED`，因此 ZCode 产物只能作为外部参考 / 临时草案。

## 7. 手动转发模式

角色未注册或工具不可用时，进入手动转发模式。

手动转发模式下，当前窗口只能输出以下内容之一：

- 给 producer tech pm 的版本计划输入摘要。
- 给 tech architect 的任务拆解请求、`PACKAGE_ASSIGNMENT` 请求或失败回流报告。
- 给 codex task writer 的任务档案写作输入，必须以上游 `PACKAGE_ASSIGNMENT` 为依据。
- 给 QA reviewer 的验收输入。
- 给 guard-agents 的边界收口摘要。
- 给 RepoOps 的版本记录输入摘要。
- 给 Cross-System Executor / ZCode 的已审批任务包执行输入或外部 QA 输入；仅当 `CROSS_SYSTEM_EXECUTOR_PROTOCOL.md` 为 `ENABLED` 时允许。当前为 `DISABLED` 时不得发送正式任务执行输入。

输出后必须停止在等待状态，不能继续冒充下一个角色完成后续流程。

## 8. QA 不通过后的强制停止点

开发窗口收到用户回复“不通过”“手测不通过”或等价失败反馈后，立即进入 QA 不通过回流状态。

开发窗口允许：

- 停止当前开发。
- 汇总失败现象。
- 整理自动 QA 与用户手测差异。
- 生成给 tech architect 的失败回流报告。
- 若 tech architect 已注册且工具可用，则真实发送报告。

开发窗口禁止：

- 自行判断最终失败归因。
- 自行重写修复任务档案。
- 自行继续 patch 或重构。
- 以“我先查一下”为理由进入代码修改。
- 在没有真实发送 / 真实返回的情况下声称已完成跨窗口流转。

开发窗口只有拿到以下五项后，才可重新获得修复开发权限：

1. 用户的不通过原因。
2. 已注册 tech architect 的正式失败归因。
3. 已注册 codex task writer 的正式修复任务档案。
4. guard-agents 对修复任务档案的明确 `GUARD_PASS`。
5. 用户对修复任务包的明确审批。

如五项不齐，开发窗口只能等待。

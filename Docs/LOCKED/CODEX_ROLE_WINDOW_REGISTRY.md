# CODEX ROLE WINDOW REGISTRY

本文件定义《符箓背包》Codex 多窗口工作流中的角色窗口注册规则。

它用于解决一个关键边界：记忆文件可以规定流程，但不能凭空完成跨窗口通信。只有角色窗口已注册、当前窗口具备真实线程读取 / 发送能力，并且消息确实发送成功时，才可称为“自动流转”。

## 1. 注册原则

- 角色窗口必须有明确角色、明确窗口标题或线程标识、明确职责范围。
- 角色窗口未注册时，不得假装已经调用该角色。
- 当前窗口无法读取或发送到目标角色窗口时，不得自行代理目标角色完成工作。
- 子代理、临时代理或当前窗口代行分析，不自动取得固定角色身份。
- 无法自动流转时，只能生成清晰报告交给用户手动转发。
- 角色注册表属于项目记忆文件，只能由唯一记忆治理窗口在用户明确审批后更新。
- 其他窗口不得自行修改本文件，不得通过声明“我是某角色”来获得注册身份。

## 3. 推荐固定窗口标题

后续如果用户创建固定角色窗口，推荐使用以下标题，便于其他窗口搜索和识别：

```text
符箓背包_Guard-agents_长期置顶
符箓背包_Producer-Tech-PM
符箓背包_Tech-Architect
符箓背包_Codex-Task-Writer
符箓背包_QA-Reviewer
```

固定窗口创建后，必须回到本长期置顶记忆治理窗口登记。登记前，其他窗口仍按“未注册”处理。

## 4. 自动流转成立条件

只有同时满足以下条件，某窗口才可以执行跨窗口自动流转：

1. 目标角色已在本文件中注册。
2. 当前窗口具备真实读取或发送目标线程的工具能力。
3. 当前窗口已经确认目标线程标题 / 标识匹配注册信息。
4. 当前窗口确实完成发送或读取，并在回复中说明结果。
5. 目标角色返回了对应产物，例如失败归因、任务档案或 QA 报告。
6. 返回产物的来源与角色注册信息一致，不是未标注来源的临时草案。

缺任一条件，均不允许声称“已调用 tech architect / codex task writer / QA reviewer”。

仅完成消息发送不代表该阶段完成；必须等待目标角色返回正式产物。

## 5. 正式产物与临时草案

- 已注册固定角色窗口真实返回的职责内产物，才可标记为正式角色产物。
- 子代理可以用于调查、列疑点、整理材料或起草建议，但必须标记为“临时分析草案”。
- 临时分析草案不得直接恢复开发权限，不得替代 architect 失败归因、task writer 修复任务档案或 Guard 回执。
- 用户明确授权某次临时代行，只授权该次辅助工作；除非本注册表随后由记忆治理窗口更新，否则不产生长期角色注册。

## 6. 手动转发模式

角色未注册或工具不可用时，进入手动转发模式。

手动转发模式下，当前窗口只能输出以下内容之一：

- 给 producer tech pm 的版本计划输入摘要。
- 给 tech architect 的任务拆解或失败回流报告。
- 给 codex task writer 的任务档案写作输入。
- 给 QA reviewer 的验收输入。
- 给 guard-agents 的边界收口摘要。

输出后必须停止在等待状态，不能继续冒充下一个角色完成后续流程。

## 7. QA 不通过后的强制停止点

开发窗口收到用户回复“不通过”“手测不通过”或等价失败反馈后，立即进入 QA 不通过回流状态。

开发窗口允许：

- 停止当前开发。
- 汇总失败现象。
- 整理自动 QA 与用户手测差异。
- 生成给 tech architect 的失败回流报告。
- 若 tech architect 已注册且工具可用，则真实发送报告。

开发窗口禁止：

- 自行判断最终失败归因。
- 自行重写修复任务档案。
- 自行继续 patch 或重构。
- 以“我先查一下”为理由进入代码修改。
- 在没有真实发送 / 真实返回的情况下声称已完成跨窗口流转。

开发窗口只有拿到以下五项后，才可重新获得修复开发权限：

1. 用户的不通过原因。
2. 已注册 tech architect 的正式失败归因。
3. 已注册 codex task writer 的正式修复任务档案。
4. guard-agents 对修复任务档案的明确 `GUARD_PASS`。
5. 用户对修复任务包的明确审批。

如五项不齐，开发窗口只能等待。

# RISK LEVEL RULE

## 绿灯

条件：

- 不触碰核心链路。
- 只修改当前版本允许的功能范围。
- 修改对象和回滚边界清晰。

例子：

- 新增首页 ComingSoon 页面
- 新增梦签入口
- 新增道藏入口
- 新增底栏图标显示
- 新增普通 UI 文案

绿灯可在完成入场审查后进入开发。

## 黄灯

条件：

- 可能影响页面流转或现有系统。
- 修改行为本身不直接触碰硬红线，但影响范围需要调查。

例子：

- 试炼入口跳转
- 战备页入口
- 页面 Root 切换
- 底栏与状态联动
- VisualAssetConfig 接入

黄灯必须先完成最小 Code Survey，明确调用链、文件范围、黄金路径 QA 和回滚方式，确认安全后才能开发。

## 红灯

条件：

- 触碰 V0.2 核心链路、稳定基线或硬红线。
- 要求重构、替换或拆分受保护系统。

例子：

- 重构 RunFlow
- 拆 PageState
- 拆 FormationState
- 替换战斗页 Root
- 修改 V02FormationGridFrame
- 修改 DamageText
- 修改奖励表
- 修改 Boss 触发
- 修改存档结构
- QA 不通过后，开发窗口绕过 tech architect 失败归因直接修复
- QA 不通过后，开发窗口绕过 codex task writer 修复任务档案直接修复
- 目标角色未在 `CODEX_ROLE_WINDOW_REGISTRY.md` 注册或不可达，却声称已完成跨窗口自动流转
- 用旧场景中的隐藏对象冒充任务要求的独立新场景或默认可见交付
- 自动测试主动激活、反射调用、临时实例化或直接切状态，为被测功能制造成功条件
- task writer 后未取得 guard-agents 的明确 `GUARD_PASS` 就开始开发
- 把子代理临时草案冒充注册角色的正式产物
- QA 不通过后，修复任务包未获用户审批就恢复开发

红灯不得直接开发。必须提交冲突报告，列出命中的锁、拟修改文件、原因、影响、风险、QA 与回滚方案，并等待用户针对该范围的明确授权。

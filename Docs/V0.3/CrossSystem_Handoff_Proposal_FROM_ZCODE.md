## 0. 一句话诉求

用户希望：当 Codex 的 token / 流量不足时，**ZCode 能接替 Codex 继续开发任务**；或长期分工——**Codex 做版本开发，ZCode 做 QA + bug 修复**。无论哪种分工，**两边必须互相知道对方干了什么、怎么干的、干得怎样**。前提：**项目记忆治理权不动，仍留 Codex**。

## 1. 用户诉求的完整复述（供 Codex 核对理解是否一致）

1. 治理权不动：`AGENTS.md` 与 `Docs/LOCKED/*` 的唯一写权限继续留在 Codex 原始治理窗口。ZCode 全程只读这些文件。
2. ZCode 作为 Codex 的"替补 / 分工执行体"存在，不是平起平坐的第二个治理者。
3. **双向感知是核心诉求**：不论哪一边完成了工作，另一边回来时必须能弄清楚——做了什么、怎么做的、结果如何。
4. 两个典型场景：
   - 场景 A：Codex token 耗尽 → ZCode 接替开发。
   - 场景 B：长期分工——Codex 开发，ZCode 做 QA + bug 修复。

## 2. 不可绕过的技术真相（物理约束）

ZCode 在调查本项目后确认以下事实，请 Codex 治理窗口核实：

1. **Codex 与 ZCode 是两套独立系统，线程级消息互通不存在。**
   - Codex 的 6 个固定角色窗口靠 Codex 内部 thread id 存在（见 `CODEX_ROLE_WINDOW_REGISTRY.md` 第 2 节注册表）。
   - ZCode **无法读取** Codex 线程内部的对话历史，**无法向** Codex 线程发送消息。
2. **唯一共享通道 = 同一个 git 仓库**（`F:\Porject\TalismanBagBrawl`）+ 磁盘文件。
3. 这与 `CODEX_ROLE_WINDOW_REGISTRY.md` 第 5 节"自动流转成立条件"一致：跨窗口流转必须**真实读取 / 真实发送并说明结果**；在跨系统场景下，"真实读取"的对象只能是**磁盘文件**而非线程消息。
4. 因此，**"共同记忆"的实质 = 双方都把工作过程与结果按约定格式落盘到同一仓库，谁回来谁读文件。** 不存在会话级互通的可能。

> 推论：本提案的全部机制都建立在"文件级共享"之上，不依赖任何线程消息能力。

## 3. 核心机制：文件级交接包（Handoff Package）

"交接包"是双方感知对方工作的**唯一物质载体**。它的字段**尽量复用 Codex 已定义的格式**，不另起炉灶：

| 感知维度 | 复用的现有格式 | 出处 |
| --- | --- | --- |
| 干了什么 | 任务档案（包体编号 / 范围 / 交付契约） | `Docs/V0.3/*_Task.md` |
| 怎么干的 | 开发回执格式（修改 / 新增 / 删除文件清单） | 任务档案第 18 节 |
| 干得怎样 | QA 报告 + 用户手测结论 | 任务档案第 12–13 节 |
| 整件事状态快照 | `PASS_SYNC_BROADCAST` 八字段 | `CODEX_VERSION_PIPELINE_LOCK.md` 第 110–119 行 |
| 可追溯证据 | `git commit` + `git diff` | 仓库本身 |

**交接包的最小构成 = 任务档案 + 开发回执 + QA 报告 + 一个 git commit**，全部为磁盘文件，Codex 与 ZCode 均可读写。

**感知方式**：回到岗位的一方，读交接包文件 + `git log`，即可还原对方做了什么。无需线程消息。

## 5. 两个场景的流转（伪流程，不落盘任何 LOCKED 改动）

### 场景 A：Codex token 耗尽 → ZCode 接替开发

```text
1. Codex 正常走流水线至：任务档案已 GUARD_PASS + 用户已审批任务包
2. 用户口头指定 ZCode 接手开发（ZCode 此时 = 当前版本开发窗口）
3. ZCode 读取磁盘上同一份任务档案，按其 NF01-00 回执 → NF01-06 开发
   （ZCode 严守任务档案的允许 / 禁止范围、测试独立性硬门）
4. ZCode 开发完成，按任务档案第 18 节格式写【开发回执】
   落盘到约定路径（建议 Docs/V0.3/<包体>_ZCode_Receipt.md）
5. 经用户授权后，ZCode 执行 git commit（commit message 标注 [ZCode]）
6. Codex 回来 → 读回执 + git log + 实际代码 diff → 还原 ZCode 做了什么
7. Codex 继续走 QA → 用户手测 → RepoOps 小版本记录 → PASS_SYNC_BROADCAST
```

### 场景 B：Codex 开发，ZCode 做 QA + bug 修复

```text
1. Codex 开发完成，产物落盘（代码 + 临时回执）
2. ZCode 读产物，按任务档案第 12 节做 QA 验收（ZCode = QA reviewer）
3. ZCode 产出【QA 报告】落盘（通过 / 不通过 + 失败原因 + 是否违反测试独立性）
4a. 通过 → Codex 读报告 → 用户手测 → 收口
4b. 不通过 → 走"不通过回流"（CODEX_VERSION_PIPELINE_LOCK 第 3 节）：
       ZCode 的失败归因在规则上只算"临时草案"
       → 必须回 Codex 的 tech architect 出正式归因
       → Codex 的 codex task writer 出正式修复档案
       → guard-agents GUARD_PASS → 用户审批修复包
       → 再由用户决定本次修复由 Codex 还是 ZCode 执行
```

## 6. 唯一合规卡点（必须由 Codex 治理窗口解决）

**问题**：ZCode 在 `CODEX_ROLE_WINDOW_REGISTRY.md` 中**没有注册**。按现行规则：

- `CODEX_ROLE_WINDOW_REGISTRY.md` 第 6 节：子代理 / 临时代理 / 当前窗口代行分析，**不自动取得固定角色身份**，其产物必须标"临时分析草案"。
- 由此推论：ZCode 产出的开发回执、QA 报告，在**字面规则下全部属于"临时草案"**，Codex 流水线不直接认，理论上还要当草案重审一遍。

这会让"接替"失去意义——ZCode 干完了，Codex 还得全套重验。

**唯一合规解法**：由 Codex 治理窗口做一次**最小修改**，在注册表（或新增一份"跨系统协议"LOCKED 文件）中定义一个新类别，例如：

- 类别名：**跨系统执行体（Cross-System Executor）**
- 身份范围：可承接"当前版本开发窗口"和"QA reviewer"
- 产物升级为"正式产物"的条件：回执/QA 报告落盘 + git commit + 用户确认
- 交接包标准格式与存放路径（建议 `Docs/V0.3/` 下，命名 `<包体>_<角色>_FROM_<系统>.md`）
- 明确 ZCode **永不获得**治理权、RepoOps 主责

> 此修改属于 LOCKED 文件变更，**只能由 Codex 原始治理窗口在用户审批后执行。ZCode 无权、也不会代为执行。**

## 8. 风险与回滚

- **风险 R1**：ZCode 误判任务边界，触碰禁止范围。
  缓解：ZCode 每次开工前强制过 `CODEX_PREFLIGHT_CHECK.md` 30 条；回执中逐项自检安全边界（参考任务档案第 18 节第 5 块）。
- **风险 R2**：交接包信息不全，Codex 回来后还原不出全貌。
  缓解：交接包强制包含开发回执 + QA 报告 + git commit hash；缺项视为交接失败，由用户裁定。
- **风险 R3**：ZCode 产物被误当正式产物，绕过 Guard。
  缓解：在注册类别生效前，ZCode 所有产物一律自标"临时草案"；类别生效后仍须经用户确认才升级为正式。
- **回滚**：本提案任何条款若导致流水线紊乱或基线风险，Codex 治理窗口可随时撤销该跨系统类别，ZCode 退回纯只读状态，不影响 V0.2 基线与任何已落盘代码。
